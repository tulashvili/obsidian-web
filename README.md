# Obsidian Secure

Веб-приложение для просмотра и редактирования Obsidian-подобного хранилища
markdown-файлов и assets. Источник истины — git-репозиторий. Локальная копия
всегда зашифрована: без мастер-пароля содержимое на диске нечитаемо.

## Ключевая идея безопасности

```
┌─ ДИСК: том ./vault-data → /data ──────── всегда зашифровано (AES-256-GCM) ─┐
│  vault-meta.json   параметры Argon2id + отпечаток ключа (не секретны)       │
│  index.enc         карта путей + имена blob'ов                              │
│  blobs/<hmac>.enc  содержимое каждого файла (имя не раскрывает путь)        │
│  repo.enc          зашифрованный tar каталога .git (история)               │
└──────────────────────────────────────────────────────────────────────────┘
                          ▲ перешифровка при изменении / синхронизации
                          │
┌─ RAM: tmpfs → /work ─────────────── plaintext, только пока контейнер жив ──┐
│  рабочее дерево git на время clone/pull/commit/push                         │
└──────────────────────────────────────────────────────────────────────────┘
```

- **Ключ выводится из мастер-пароля** (Argon2id) и существует **только в
  оперативной памяти**. На диск ключ и пароль не пишутся никогда.
- На постоянном томе `/data` лежит **только зашифрованное**. Получив доступ к
  диску компьютера, без пароля прочитать ничего нельзя.
- Расшифрованные данные живут в **tmpfs** (`/work`, в RAM) только во время
  git-операций и отдаются клиенту по запросу из памяти. При остановке
  контейнера tmpfs исчезает — plaintext не остаётся на диске.
- **Имена blob'ов** — это `HMAC(путь)`, поэтому даже структура каталогов на
  диске не раскрывается.

> ⚠️ Пока контейнер запущен и разблокирован, расшифрованные данные находятся в
> RAM процесса. Защита направлена против доступа к **диску в выключенном/
> заблокированном состоянии**, а не против атаки на работающий процесс с правами
> root. Это соответствует исходному требованию.

## Стек

- **Бэкенд:** Go (один статический бинарник, быстрый на тысячах файлов).
- **Крипто:** `golang.org/x/crypto/argon2` + AES-256-GCM (stdlib).
- **Markdown:** goldmark (GFM) + собственная обработка `[[wiki-ссылок]]`.
- **Git:** системный `git` (надёжный clone/pull --rebase/push, авторизация).
- **Фронтенд:** ванильный SPA, встроен в бинарник; редактор — CodeMirror 6
  (бандл собирается заранее в `cmd/server/web/cm.js`).

## Возможности

- Просмотр markdown с рендером в HTML и assets (картинки, pdf, видео, аудио).
- **Wiki-ссылки** `[[Заметка]]`, `[[Заметка|алиас]]`, `[[Заметка#заголовок]]`,
  встраивание `![[image.png]]`.
- **Live-preview редактор** (CodeMirror 6, как в Obsidian): заголовки/жирный/
  курсив/код рендерятся «по месту», а служебные символы скрываются, пока курсор
  не на них; `[[ссылки]]` — кликабельные, `![[картинки]]` — встроенные.
- **Автодополнение `[[`** по списку заметок прямо в редакторе.
- **Создание новой заметки** (кнопка «＋ Заметка» в боковой панели).
- **Обратные ссылки** (backlinks) с контекстом.
- Поиск по именам и содержимому.
- Редактирование с коммитом и push обратно в git (кнопка «Обновить»).
- Дерево файлов с каталогами.

## Запуск

1. Подготовьте конфиги:
   ```bash
   cp .env.example .env
   cp docker-compose.example.yml docker-compose.yml
   # заполните .env: впишите OSA_REPO_URL и при необходимости токен/SSH-ключ
   ```
2. Соберите и запустите:
   ```bash
   docker compose up --build
   ```
3. Откройте <http://localhost:8080>. При **первом** запуске введённый пароль
   становится мастер-паролем хранилища (запомните его — восстановления нет).
4. Нажмите **«⟳ Обновить»** — приложение клонирует репозиторий, зашифрует файлы
   в `./vault-data` и построит граф ссылок.

При последующих запусках вводите тот же пароль; неверный пароль не
расшифрует хранилище.

> `docker compose up --build` **не требует Node** — бандл редактора
> `cmd/server/web/cm.js` уже собран и встроен в бинарник через `go:embed`.

### Пересборка редактора (CodeMirror)

Нужна, только если вы меняли исходники редактора в `editor/src/`:

```bash
cd editor
npm install      # один раз
npm run build    # esbuild → ../cmd/server/web/cm.js
```

После этого пересоберите бинарник (`docker compose up --build`).

## Будущая авторизация

Архитектура уже готова к ней: мастер-пароль фактически и есть авторизация —
без него ключа нет, а значит нет и доступа к данным. Когда решите, как именно
делать вход (логин/пароль, OIDC, passkey), нужно лишь связать механизм входа с
получением мастер-пароля/ключа (например, расшифровывать ключ хранилища
ключом, привязанным к пользователю). Никаких изменений в формате хранения не
потребуется.

## Ограничения MVP / что дальше

- Граф ссылок пересобирается целиком при сохранении файла — для очень частых
  правок стоит сделать инкрементальное обновление.
- Большие assets грузятся в память целиком при отдаче — для огромных файлов
  имеет смысл потоковое (chunked) шифрование.
- Конфликты при `git pull --rebase` сейчас просто сообщаются ошибкой —
  разрешение конфликтов в UI не реализовано.
- Нет графа-визуализации (graph view) и tag-навигации — легко добавить поверх
  уже имеющегося индекса.

## Структура проекта

```
cmd/server/          точка входа + встроенный фронтенд (web/, вкл. cm.js)
editor/              исходники редактора CodeMirror 6 + сборка (esbuild)
internal/config/     конфигурация из переменных окружения
internal/crypto/     Argon2id KDF + AES-256-GCM (ядро безопасности)
internal/store/      зашифрованное хранилище на /data
internal/index/      граф wiki-ссылок и backlinks (в RAM)
internal/render/     markdown → HTML с wiki-ссылками
internal/gitsync/    синхронизация с git через tmpfs
internal/api/        HTTP API + экран разблокировки
```
